# 🌳 Study AI with Med - 项目架构说明

本项目不仅是一个静态博客，而是基于 **Next.js 16+ (App Router)** 打造的可交互医学 AI 研究图谱。它融合了动态路由、MDX 富文本解析引擎、WASM Python 浏览器运行时，以及统一的语义与 JSON 暴露层。

下面是项目的完整代码结构深度解析：

## 🏗️ 架构核心概念

### 数据流

```
content/*.mdx → lib/mdx.ts (gray-matter 解析) → lib/navigation.ts (目录树构建)
                                              ↓
                          research/[...slug]/page.tsx (SSR 渲染)
                                              ↓
                          MDXRemote(sharedMdxOptions) → components/mdx/* (拦截器)
```

### 内容目录约定

每个研究方向（`ai-{topic}/`）遵循统一的目录结构约定：

| 文件 | 用途 | 必需 |
|------|------|------|
| `intro.mdx` | 简单理解入门文章（`order: 0`） | ✅ 是 |
| `industry-overview.mdx` | 行业全景深度报告（`order: 1`） | ✅ 是 |
| `tools/*.mdx` | 工具部署指南与实操教程（`order: 2+`） | ❌ 可选 |

> ⚠️ **导航扁平化注意**：`tools/` 子目录下的文章在侧边栏中会被**扁平显示**在所属方向分组下，不会形成子分组。这是 `navigation.ts` 按第一段路径进行分组的设计决策。

### 辅助区域约定

| 目录 | 导航标题 | 排序 | 说明 |
|------|---------|------|------|
| `knowledge/` | 知识点分析 | 10 | 基础概念、术语解读、通识知识 |
| `notes/` | 个人笔记区 | 80 | 作者的学习笔记与思考碎片 |
| `forum/` | 微论坛 | 90 | 社区交流空间（规划中） |

> 📌 **与后端规划的关系**：`后端功能规划.md` 中的 Phase 2 用户笔记系统（`/newnote/`）使用数据库（Supabase）存储用户提交内容，与 `content/notes/` 中的**作者个人笔记**（Git 管理）是两个独立概念。前者是 UGC，后者是原创内容。

### 共享模块设计

| 模块 | 位置 | 用途 |
|------|------|------|
| `sharedMdxOptions` | `lib/mdx.ts` | remark/rehype 插件链的单一真相源 |
| `useScrollSpy` | `hooks/useScrollSpy.ts` | IntersectionObserver + readProgress 统一逻辑 |
| `useScrollToHeading` | `hooks/useScrollSpy.ts` | 平滑跳转 + URL Hash 更新 |
| `useBodyScrollLock` | `hooks/useScrollSpy.ts` | Modal/Drawer 的 body 滚动锁定 |
| `buildNavigationTree` | `lib/navigation.ts` | content/ 目录 → 侧边栏树 |

## 🛠️ 设计哲学摘要

1. **绝对的内容解耦：** 作者只需要在 `content/` 按照约定的目录结构写内容，所有路由、侧边栏和目录表完全依靠 `lib/navigation.ts` 动态扫描编译期搞定，完全零配置。
2. **统一命名范式：** 所有研究方向使用 `ai-{topic}` 前缀统一命名，每个方向内部遵循 `intro → industry-overview → tools/` 的固定结构约定。
3. **"真"交互探索：** 脱离图文束缚。读者通过 `<PythonRunner>` 在前沿学术文档里直接用 NumPy；通过 `<MermaidChart>` 读取动态流程状态。
4. **彻底的 AI 原生：** 你不仅可以作为一个人类在这里阅读，这套代码提供了 `/api/content` 作为数字人的大门。任何外部的 LLM AI Agent 都可以结构化调取它需要的文章 JSON 进行精读总结。
5. **DRY 原则：** 所有跨组件复用的逻辑（Scroll Spy、MDX 插件配置）集中到 `hooks/` 和 `lib/` 中的共享模块，杜绝代码重复。
