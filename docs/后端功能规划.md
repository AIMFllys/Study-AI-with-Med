## 1. 项目现状基线

### 当前技术栈

| 维度 | 状态 |
|------|------|
| 框架 | Next.js 16.1.6 (App Router) |
| 前端 | React 19.2 + Tailwind v4 (CSS-first) + TypeScript 5 |
| 内容系统 | MDX 文件 → `next-mdx-remote` SSR 渲染 |
| API 层 | 仅 1 个 Route Handler (`api/content/route.ts`) |
| 数据库 | ❌ 无 |
| 认证系统 | ❌ 无 |
| 端口 | 5986 |
| 包管理 | npm |

### 现有路由地图

```
/                        ← Landing Page
/research                ← 研究总览（Sidebar + MDX 渲染）
/research/[...slug]      ← 动态文章（通配路由）
/about                   ← 关于页面
/playground              ← Python 实验场
/api/content             ← MDX 元数据 JSON API
```

### 核心设计约束

1. **内容解耦**: 原始学术文章在 `content/` 目录以 MDX 文件存管，由 Git 版本控制
2. **用户 UGC 分离**: 所有用户生成内容（评论/笔记/修订）存入 Supabase 数据库
3. **无用户登录**: 本项目不引入用户注册/登录体系，交互功能采用匿名模式
4. **后端内置**: 使用 Next.js Route Handlers，前后端同进程、同端口 (5986)

## 2. 总体架构设计

### 架构全景图

```
┌──────────────────────────────────────────────────────────────────────┐
│                      Next.js 16 App Router (端口 5986)               │
├──────────────┬─────────────────┬─────────────────────────────────────┤
│   现有前端     │    新增前端页面    │          新增 Route Handlers         │
│ ──────────── │ ─────────────── │ ─────────────────────────────────── │
│ /            │ /newnote/[id]   │ GET|POST  /api/comments             │
│ /research/*  │ /newnote/submit │ GET|POST  /api/notes                │
│ /about       │ /fix/[id]       │ GET|POST  /api/revisions            │
│ /playground  │ /fix/submit     │ POST      /api/ai/chat     (SSE)   │
│              │ /admin/*        │ POST      /api/ai/explain           │
│              │                 │ PATCH|DEL /api/admin/*              │
│              │                 │ POST      /api/admin/auth           │
├──────────────┴─────────────────┴─────────────────────────────────────┤
│                          公共后端工具层                                │
│    src/lib/supabase.ts  ·  src/lib/admin-auth.ts  ·  src/lib/rate-limit.ts │
├──────────────────────────────────────────────────────────────────────┤
│                    @supabase/supabase-js (service_role)               │
├──────────────────────────────────────────────────────────────────────┤
│                  Supabase (1037SoloPlatform 共享实例)                  │
│              medicine_comments | medicine_user_notes | ...            │
│                    + RLS Policies + Table Comments                    │
└──────────────────────────────────────────────────────────────────────┘
         │                              │
         ▼                              ▼
   DeepSeek / OpenAI API         GitHub API (可选)
   (AI 对话 & 划词解释)           (修订自动 PR)
```

### 后端技术选型理由

| 对比维度 | Next.js Route Handlers ✅ | Express.js ❌ | Python ❌ |
|---------|--------------------------|-------------|----------|
| 开发效率 | 零配置即可开始 | 需搭建独立骨架 | 需 venv + pip |
| 运行方式 | 同端口 (5986) 单进程 | 双进程 + CORS | 跨语言 + 独立运行时 |
| 部署 | `next start` 一条命令 | 需进程管理工具 | 需 Python 运行时 |
| Supabase SDK | 官方 JS SDK | 官方 JS SDK | 社区 supabase-py |
| 项目 API 量级 | ~15 路由，完全胜任 | 杀鸡用牛刀 | 过度复杂 |

详细分析参考选型文档：Express.js vs Route Handlers 选型分析。

### 3.2 表清单与 Schema

#### `medicine_comments` — 文章评论

```sql
CREATE TABLE medicine_comments (
  id              UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  article_slug    TEXT NOT NULL,                    -- 关联的文章 slug
  nickname        TEXT NOT NULL DEFAULT '匿名用户',  -- 用户昵称
  content         TEXT NOT NULL,                    -- 评论内容
  status          TEXT NOT NULL DEFAULT 'pending',  -- pending | approved | rejected
  ip_address      INET,                            -- 提交者 IP（频率限制用）
  created_at      TIMESTAMPTZ DEFAULT now()
);

ALTER TABLE medicine_comments ENABLE ROW LEVEL SECURITY;

COMMENT ON TABLE medicine_comments IS
  '子项目 medicine (Study-AI-with-Med) 的文章评论表';
```

**RLS 策略**:
- `SELECT`: 公开可读 `status = 'approved'` 的评论
- `INSERT`: 公开可写（anonymousRole 或 service_role）
- `UPDATE/DELETE`: 仅 `service_role`（管理端 Route Handler 使用）

### Phase 1: 评论系统 + 后台管理面板

> **优先级**: P0  
> **预估工时**: 3-4h

#### 1.1 用户端：文章底部评论区

**位置**: 嵌入现有 `research/[...slug]/page.tsx` 页面底部

**组件**:
- `<CommentSection slug={slug} />` — 评论列表 + 提交表单

**交互流程**:
```
用户在文章底部看到评论区
→ 填写昵称（可选，默认「匿名用户」）+ 评论内容
→ 点击「发送」
→ POST /api/comments { slug, nickname, content }
→ 后端校验 + IP 频率限制 + 存入 DB (status='pending')
→ 前端提示「评论已提交，等待审核后显示」
```

**API 路由**:

| 方法 | 路径 | 用途 |
|------|------|------|
| `GET` | `/api/comments?slug=xxx` | 获取该文章已审核的评论列表 |
| `POST` | `/api/comments` | 提交新评论（默认 pending） |

#### 1.2 管理端：后台审核面板

**路由**: `/admin` （独立页面，密码保护）

**登录流程**:
```
管理员访问 /admin
→ 输入管理密码（对比环境变量 ADMIN_PASSWORD）
→ 密码存入 sessionStorage（仅当前标签页有效）
→ 后续所有 admin API 请求携带 x-admin-password Header
```

**管理 API 路由**:

| 方法 | 路径 | 用途 |
|------|------|------|
| `POST` | `/api/admin/auth` | 验证管理密码 |
| `GET` | `/api/admin/comments` | 获取所有评论（含 pending） |
| `PATCH` | `/api/admin/comments/[id]` | 审核评论 (approved/rejected) |
| `DELETE` | `/api/admin/comments/[id]` | 删除评论 |

**面板页面**:

| 页面 | 路径 | 功能 |
|------|------|------|
| 管理首页 | `/admin` | 数据概览仪表板（各模块待审核数量） |
| 评论管理 | `/admin/comments` | 列表 + 审核/删除操作 |

### Phase 2: 用户笔记提交系统

> **优先级**: P1  
> **预估工时**: 4-5h  
> **动态路由**: `/newnote/[noteId]` （格式: `YYYYMMDD-shortId`）

#### 2.1 提交流程

```
用户访问 /newnote/submit
→ 填写：标题 + 作者昵称（可选）+ Markdown 内容
→ 实时预览 Markdown 渲染效果
→ 点击「提交笔记」
→ POST /api/notes { title, author_name, content_md }
→ 后端生成 noteId = `${YYYYMMDD}-${随机6位}`
→ 存入 medicine_user_notes (status='pending')
→ 前端提示「笔记已提交，审核通过后可在 /newnote/XXXXXX-XXX 查看」
```

#### 2.2 查看页面

```
用户或管理员访问 /newnote/20260314-a1b2c3
→ 前端页面 SSR 调用 Supabase 查询该 noteId
→ 若 status='approved'，渲染 Markdown 内容
→ 若 status='pending'，显示「审核中」提示
→ 若不存在，显示 404
```

#### 2.3 API 路由

| 方法 | 路径 | 用途 |
|------|------|------|
| `GET` | `/api/notes` | 获取已审核笔记列表 |
| `POST` | `/api/notes` | 提交新笔记 |
| `GET` | `/api/notes/[noteId]` | 获取单条笔记详情 |

#### 2.4 管理端

| 页面 | 路径 | 功能 |
|------|------|------|
| 笔记管理 | `/admin/notes` | 审核/删除用户笔记 |

**管理 API**:

| 方法 | 路径 | 用途 |
|------|------|------|
| `GET` | `/api/admin/notes` | 获取所有笔记（含 pending） |
| `PATCH` | `/api/admin/notes/[id]` | 审核笔记 |
| `DELETE` | `/api/admin/notes/[id]` | 删除笔记 |

#### 2.5 Markdown 编辑器方案

| 方案 | 依赖 | 优缺点 |
|------|------|--------|
| **A: textarea + 预览** (推荐) | `react-markdown` | 轻量、零额外依赖、可自定义 |
| B: @uiw/react-md-editor | `@uiw/react-md-editor` | 功能丰富但体积较大 |

推荐方案 A：左侧 `<textarea>` 输入，右侧 `react-markdown` 实时预览。

### Phase 3: AI 对话 + 划词解释

> **优先级**: P1  
> **预估工时**: 3-4h  
> **前置条件**: 需要 AI API Key（DeepSeek / OpenAI / 通义千问）

#### 3.1 AI 对话窗

**组件**: `<AIChatPanel />` — 悬浮在页面右下角

**交互流程**:
```
用户点击右下角 AI 图标 → 展开对话面板
→ 输入问题：「AlphaFold3 的扩散模型原理是什么？」
→ POST /api/ai/chat { message, context[] }
→ Route Handler 调用 AI API（流式 SSE）
→ 前端逐字显示 AI 回复（打字机效果）
```

**API**:

| 方法 | 路径 | 响应类型 | 用途 |
|------|------|---------|------|
| `POST` | `/api/ai/chat` | `text/event-stream` (SSE) | 流式对话 |

**System Prompt 示例**:
```
你是一位 AI 赋能医学领域的研究助手。用户正在阅读一个关于 AI 与医学交叉
领域的研究笔记网站。请用通俗易懂的中文回答他们的问题，必要时引用具体论文
或技术细节。保持专业但友好的语气。
```

**上下文策略**:
- 携带当前页面的文章 slug，系统自动加载文章摘要作为背景知识
- 对话保留最近 **5 轮** 上下文，不做长期记忆
- 单次回复限制 `max_tokens: 1000`

#### 3.2 划词 AI 解释

**组件**: `<TextSelectionPopover />` — 全局挂载

**交互流程**:
```
用户在文章中选中一段文字（如 "条件扩散模型"）
→ 鼠标抬起时检测选区
→ 弹出浮动按钮「🤖 AI 解释」
→ 用户点击 → POST /api/ai/explain { text, article_slug }
→ AI 返回简短解释（非流式，一次性返回）
→ 在气泡中展示解释结果
```

**API**:

| 方法 | 路径 | 用途 |
|------|------|------|
| `POST` | `/api/ai/explain` | 划词解释（非流式，max_tokens=300） |

**Explain System Prompt 示例**:
```
用户在一篇关于 AI 赋能医学的学术文章中选中了一段文字。
请用 2-3 句话简明扼要地解释这个概念或术语，适合医学+CS 交叉领域的初学者阅读。
```

#### 3.3 费用控制策略

| 控制点 | 措施 |
|--------|------|
| 单次对话 | `max_tokens: 1000` |
| 单次解释 | `max_tokens: 300` |
| IP 频率 | 每 IP 每分钟最多 5 次 AI 请求 |
| 每日上限 | 环境变量 `AI_DAILY_LIMIT` 控制全局每日总量 |

### Phase 4: 百科式版本修订

> **优先级**: P2  
> **预估工时**: 4-5h  
> **动态路由**: `/fix/[revisionId]` （格式: `YYYYMMDD-shortId`）

#### 4.1 提交修订

```
用户在文章页面点击「提交修订」按钮
→ 跳转到 /fix/submit?slug=protein-engineering
→ 编辑器预加载原文内容（从 /api/content?slug=xxx 获取）
→ 用户修改内容 + 填写修订说明
→ POST /api/revisions { target_slug, revised_content, revision_note }
→ 后端生成 revisionId，存入 DB (status='pending')
→ 提示「修订已提交，等待审核」
```

#### 4.2 修订审核与对比

**查看页面**: `/fix/20260314-a1b2c3`

```
展示两栏对比视图：
┌──────────────────┬──────────────────┐
│     原文 📄       │    修订后 ✏️       │
│                  │                  │
│  高亮 删除内容     │  高亮 新增内容     │
│  (红色背景)       │  (绿色背景)       │
│                  │                  │
└──────────────────┴──────────────────┘
```

**Diff 对比**: 使用 `react-diff-viewer-continued` 组件

#### 4.3 合并策略

审核通过的修订不直接修改 MDX 文件（保持 Git 管理），而是：

**方案 A (推荐)**: 修订以「社区贡献」板块展示在文章底部

```
┌─────────────────────────────┐
│  📄 原文内容（MDX 渲染）      │
├─────────────────────────────┤
│  💬 评论区                   │
├─────────────────────────────┤
│  ✏️ 社区修订贡献              │  ← 从 DB 读取 approved 修订
│  - 修订 #1 by 张三 (2026-03) │
│  - 修订 #2 by 李四 (2026-04) │
└─────────────────────────────┘
```

**方案 B (远期)**: 管理员审核通过后，通过 GitHub API 自动创建 PR

#### 4.4 API 路由

| 方法 | 路径 | 用途 |
|------|------|------|
| `GET` | `/api/revisions?slug=xxx` | 获取文章的已审核修订 |
| `POST` | `/api/revisions` | 提交修订 |
| `GET` | `/api/revisions/[revisionId]` | 获取单条修订详情 |

**管理 API**:

| 方法 | 路径 | 用途 |
|------|------|------|
| `PATCH` | `/api/admin/revisions/[id]` | 审核修订 |
| `DELETE` | `/api/admin/revisions/[id]` | 删除修订 |

## 7. 安全策略

### 7.1 管理认证

| 层级 | 机制 |
|------|------|
| 认证方式 | 环境变量 `ADMIN_PASSWORD` 简单密码 |
| 携带方式 | 请求 Header `x-admin-password` |
| 有效范围 | 每个 admin API 请求独立验证 |
| 暴力破解防护 | Rate Limiting (每 IP 每分钟 3 次管理 API 调用) |

### 7.2 反垃圾

| 维度 | 策略 |
|------|------|
| IP 频率限制 | 每 IP 每分钟最多 5 次评论/笔记/修订提交 |
| 内容限制 | 评论 ≤ 500 字，笔记 ≤ 10000 字，修订 ≤ 20000 字 |
| 蜜罐字段 | 表单中隐藏一个 `website` 字段，填写则视为机器人 |
| 内容校验 | 去除首尾空白，不允许纯空格内容 |

### 7.3 XSS 防护

- 所有用户提交的 Markdown 渲染前经过 `sanitize-html` 过滤
- 禁止 `<script>`、`<iframe>`、`<style>` 等危险标签
- 渲染使用 `react-markdown`（默认安全，不执行原始 HTML）

### 7.4 API Key 保护

- AI API Key 仅在 Route Handlers（服务端）中使用
- 通过环境变量注入，不在前端代码中出现
- 环境变量文件 `.env.local` 已在 `.gitignore` 中排除

## 9. 风险与应对

### 🔴 高风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 无用户认证体系 | 所有用户交互功能缺乏身份基础 | 匿名 + 昵称模式；后台用环境变量密码 |
| AI API Key 泄露 | 被恶意调用产生费用 | 全部走 Route Handlers 服务端；.env.local 禁止提交 |
| 评论/笔记被恶意刷量 | 数据库膨胀 + 管理负担 | IP 频率限制 + 蜜罐字段 + 内容长度限制 |

### 🟡 中风险

| 风险 | 影响 | 应对 |
|------|------|------|
| MDX 文件 vs 数据库双源 | 数据管理复杂度 | 严格分离：原创内容 ← 文件系统；UGC ← 数据库 |
| 管理密码暴力破解 | 后台被入侵 | Rate Limiting + 后续可升级 NextAuth.js |
| AI Token 费用失控 | 月度超额账单 | 每日上限 + 单次 max_tokens 限制 |

### 🟢 低风险

| 风险 | 影响 | 应对 |
|------|------|------|
| Supabase 免费额度耗尽 | 服务不可用 | 复用现有 `1037SoloPlatform`，medicine_ 前缀零额外费用 |
| Route Handler 性能瓶颈 | 高并发时响应慢 | 当前访问量级不会触发；真到瓶颈时迁移 Express.js |
